## 概述

《明日方舟：终末地》是鹰角网络开发的3D即时策略RPG，故事发生在天灾横行的塔卫二星球。游戏融合了即时战斗、基地建设（集成工业）和箱庭世界探索，采用**四人小队同屏作战**的核心玩法——玩家主控一名角色，其余三名由AI辅助控制，并可随时切换。

终末地的相机系统面临一个独特挑战：它既不是纯粹的动作游戏（如鬼泣、战神），也不是传统的开放世界探索游戏（如原神）。它需要同时服务于**策略性的团队指令战斗**和**工业化箱庭世界的深度探索**——这两种截然不同的体验对相机提出了不同的要求。

::ImageView{src="/images/placeholder/endfield-overview.jpg" caption="终末地的箱庭世界：工业美学与自然景观的融合" width="full"}
::

### 核心设计理念
- **自由TPS为主**：标准第三人称跟随视角，玩家可完全自由控制相机旋转
- **策略战斗适配**：战斗中相机需同时兼顾主控角色和多名同屏队友的可见性，而非单纯追踪一个角色
- **多场景适配**：探索、战斗、工业建造、Boss战、演出等不同场景下相机行为有明显差异
- **跨平台兼顾**：PC与移动端双端适配，相机操控需适应手柄/键鼠/触屏三种输入方式

### 为什么研究这款游戏的相机

终末地的相机设计有几个值得关注的研究点：

1. **多角色同屏的构图挑战**：四人同屏作战时，相机如何确保主控角色清晰可见，同时不丢失队友和敌人的信息？这与单主角游戏（战神、鬼泣）有本质区别
2. **策略与动作的张力**：游戏本质是"即时指令战斗"而非纯动作游戏，相机需要服务于技能释放的判断而非操作反应
3. **箱庭世界的垂直空间**：终末地的地图设计包含大量垂直结构（矿洞、高台、管道网络），相机碰撞和视角适配面临特殊挑战
4. **工业建造视角**：集成工业系统中涉及大量俯视/规划类操作，需要一套独立于战斗的相机逻辑

## 探索相机

### 跟随逻辑

终末地的探索相机采用标准的第三人称弹簧臂跟随方案，角色位于画面偏左约1/3处，右侧留出较大的探索视野空间。

- 弹簧臂跟随距离中等（约3.5-4m），比战神远但比原神略近，在近距离沉浸感和视野范围之间取了平衡
- 角色移动时相机有轻度惯性延迟（约0.2-0.3s），急停和转向时能感受到相机的"追赶"
- 角色静止不动时相机保持稳定，无明显的呼吸或漂浮效果
- 在队伍其他成员跟随行走时，相机仅追踪主控角色，不会因AI队友的移动而调整

::ImageView{src="/images/placeholder/endfield-explore-follow.jpg" caption="探索视角下的跟随距离与角色位置：标准TPS偏左构图" width="full"}
::

### 视角操控

- 右摇杆/鼠标控制水平+垂直旋转，操控范围自由
- 垂直旋转有上下限位角度限制，防止穿入地面或翻转到角色头顶
- 移动端触屏操作通过右侧屏幕区域的滑动控制视角旋转，灵敏度与手柄/键鼠独立调节
- 探索状态下**存在弱自动回正**：长时间不操作相机（约5-8秒），相机会缓慢回正到角色身后偏上的默认位置
- 角色进行冲刺（sprint）时，相机会略微拉远约0.3-0.5m，增加速度感并扩大前方视野

### 碰撞处理

- 相机碰到墙壁、岩石等障碍物时会向角色方向推近，整体表现中规中矩
- 碰撞推近的过渡有基本的缓动处理，不会产生突然的跳切
- 离开碰撞区域后恢复原始距离有轻微延迟（约0.3-0.5s）
- 在部分场景（如矿洞转角、狭窄的工业管道区域）碰撞处理偶尔出现穿模或"贴脸"现象

### 室内/狭窄空间

终末地的箱庭世界包含大量室内和狭窄空间——矿洞、地下设施、工业管道区域等，相机对这些场景做了一定的适配：

- 进入狭窄走廊/洞穴时相机自动拉近距离，同时FOV略微增大以补偿视野
- 部分特殊空间（如过道、电梯区域）使用了预设相机引导，限制旋转自由度以避免穿墙
- 在密集工业建筑区域（玩家自建的传送带、钻机等），相机碰撞检测面临额外挑战——自建结构物的碰撞体并非总是完善

::ImageView{src="/images/placeholder/endfield-indoor-camera.jpg" caption="地下矿洞中的相机表现：自动拉近距离并轻微增大FOV" width="full"}
::

### 工业建造视角

这是终末地独有的相机模式，在进入集成工业系统的建造/编辑界面时生效：

- 相机从角色跟随视角切换为**半俯视的自由视角**，可自由平移、旋转和缩放
- 缩放范围较大，从近距离细节查看到较远距离的全局规划俯瞰均支持
- 此模式下相机操控逻辑更接近RTS/模拟经营类游戏，与战斗/探索的TPS操控形成明显区分
- 退出建造界面时，相机通过一次平滑过渡（约0.5-0.8s）回到角色跟随视角

## 战斗相机

战斗相机是终末地相机系统中最核心也最复杂的部分。由于游戏采用**四人同屏即时战斗**的机制，且战斗定位为"注重策略的即时战斗"（而非纯动作），相机需要在多方面做出权衡。

### 战斗状态切换

进入战斗状态时，相机会产生一系列参数调整：

- 相机距离从探索的~3.5-4m拉远至~4.5-5m，以容纳更多角色和敌人
- FOV略微增大（约+3-5°），进一步扩展水平视野
- 相机高度偏移略微增加，让视角稍偏俯视，有利于观察地面范围指示器（Boss的"地板技"AOE预警等）
- 跟随延迟减小，相机对角色移动的响应更为即时
- 过渡过程平滑（约0.3-0.5s），不会产生突兀的镜头跳切

::VideoEmbed{src="/videos/placeholder/endfield-combat-transition.mp4" caption="从探索状态进入战斗时的相机参数过渡"}
::

### 多角色同屏构图

这是终末地相机设计中最独特的课题。四名角色同时出现在战场上，相机需要解决：

**主控角色优先**：
- 相机始终以当前主控角色为跟踪目标，构图以主控角色为核心
- 主控角色在画面中保持相对稳定的屏幕位置（偏左约1/3），不会因队友移动而偏移

**队友可见性处理**：
- AI控制的队友会**自动调整站位**以避免遮挡主控角色的视线，这是一个巧妙的设计——不是通过相机适配来解决遮挡问题，而是让角色本身规避遮挡
- 当主控角色持有爆炸物等危险品靠近队友时，队友会主动远离，这一行为在客观上也减少了角色重叠造成的视觉混乱
- 在群战中，相机**不会主动拉远到容纳所有人的距离**——它选择信任主控角色视角，而非试图展现全局

**敌人信息传达**：
- 大量依赖UI元素（伤害数字、状态图标、Boss血条等）来补偿相机视野之外的敌人信息
- Boss战中，Boss的AOE技能通过地面红圈等明确的视觉提示传达，不强制要求相机看到Boss本体
- 战斗中屏幕边缘会出现**伤害方向指示**，引导玩家转向受到攻击的方向

### 锁定系统

终末地在近期版本中对锁定系统做了简化设计：

- 提供对单体敌人的**软锁定辅助**：靠近敌人时攻击会自动朝向最近目标
- 不设置传统的硬锁定按钮（如DMC的R1/战神的R3），这与游戏"策略指令"而非"精确动作"的定位一致
- 软锁定的影响范围有限——主要影响普攻和部分定向技能的朝向，不会改变相机行为
- 面对多个敌人时，攻击方向由**玩家摇杆/移动方向输入**决定，而非锁定目标自动朝向

这一设计选择意味着终末地**放弃了锁定相机的概念**，相机始终保持自由旋转状态。这有利有弊——减少了相机行为的复杂度，但也缺少了锁定视角对战斗节奏的引导。

### 非锁定状态

由于没有硬锁定，非锁定就是终末地战斗的**唯一常态**：

- 相机朝向由玩家自由控制，战斗中不存在自动朝向敌人的相机行为
- 角色的攻击方向由摇杆输入决定，**相机朝向即大致的攻击方向**（类似战神的越肩设计思路）
- 群战时需要玩家手动转动相机来获取不同方向的信息
- 由于战斗核心是技能释放而非精确操作，对相机自由操控的依赖程度低于纯动作游戏

### 目标切换

由于没有硬锁定系统，终末地不存在传统意义上的"锁定目标切换"：

- 多目标场景下，通过**移动方向+相机朝向**来隐式选择攻击对象
- 部分技能带有自动追踪或范围效果，降低了对精确目标选择的要求
- Boss战中通常只有一个核心目标，目标切换的需求场景有限
- 召唤物（Boss的小怪阶段）通过UI标记区分优先级

### 闪避与相机

终末地设有闪避机制（完美闪避可获得无敌帧），闪避时相机的表现：

- 闪避动作期间相机继续跟随角色移动，无特殊的慢动作或zoom效果
- **完美闪避不触发子弹时间**——这是一个有意的设计选择，保持了战斗的即时策略节奏，而非鼓励玩家追求精确操作奖励
- 冲刺闪避时相机跟随延迟与正常移动一致，不做特殊加速或减速处理
- 这一设计与游戏"策略优先于操作"的整体理念保持一致

### 镜头抖动与反馈

终末地的相机反馈相比纯动作游戏偏克制：

- 普攻命中时有**极轻微的camera shake**，幅度小于DMC5和战神
- 技能命中时shake幅度略有增加，但整体仍然偏轻
- **连携技触发时是反馈最强的时刻**——画面会有短暂的顿帧（hit-stop）配合轻度shake和zoom-in，强调连携成功的爽快感
- Boss的大范围攻击命中玩家时有较明显的camera shake，作为受创反馈
- 整体反馈风格偏向"干净利落"而非"沉重有力"，符合游戏策略RPG而非动作游戏的定位

## 特殊相机

### 过场演出

终末地的叙事过场采用较为标准的实时渲染方案：

- 剧情对话使用实时渲染的角色演出，相机切换到预设的对话构图（正反打、中景等）
- 重要剧情节点使用更精心编排的镜头语言——推拉、摇移、跟拍等
- 过场与游玩之间**非一镜到底**，存在明确的镜头切换/黑屏过渡
- 进入过场时UI元素淡出，结束时淡入恢复，过渡过程约0.3-0.5s
- 部分对话支持跳过，跳过后相机直接切回游玩视角

::ImageView{src="/images/placeholder/endfield-cutscene.jpg" caption="剧情过场的实时渲染演出：标准的对话正反打构图" width="full"}
::

### 终结技演出

终结技（大招）是终末地中最具视觉冲击力的相机演出时刻：

- 释放终结技时，游戏进入**短暂的演出模式**——时间暂停（或极度减缓），相机从游玩视角切换到预设的演出轨道
- 每个角色的终结技有**独立设计的镜头编排**：包含特写推入、动态环绕、低角度仰拍等手法
- 终结技演出期间角色有专属的全屏动画表现，背景适当虚化以突出角色
- 演出时长控制在2-4秒左右，结束后相机通过一次快速但平滑的过渡切回战斗视角
- 多次观看后演出不可跳过但时长较短，对战斗节奏的打断控制在可接受范围

::VideoEmbed{src="/videos/placeholder/endfield-ultimate-camera.mp4" caption="终结技演出：预设镜头轨道的角色特写与动态环绕"}
::

### 连携技演出

连携技的相机演出比终结技更为克制：

- 连携技触发时有**短暂的镜头特写**（约0.5-1秒），快速zoom-in到释放连携技的角色
- 不会完全暂停时间，而是短暂的慢动作（hit-stop效果）
- 由于连携技在高强度战斗中可能频繁触发，演出被刻意设计得简短以避免打断战斗流
- 多个连携技连续触发时，演出节奏形成一种"连珠炮"式的快感——这是终末地战斗"倾泻感"的核心来源

### Boss战相机

Boss战中相机有一些针对性的调整：

- 面对大型Boss时，相机会**自动拉远**更多距离，确保Boss的关键攻击前摇和AOE范围在视野内
- Boss的蓄力攻击阶段，地面会出现红色/橙色的范围指示器，相机高度偏移增加以确保地面指示清晰可见
- Boss的相位转换（如切换攻击模式、召唤小怪等）通常伴随短暂的镜头演出——拉远展示Boss形态变化或场景变化
- 部分Boss战（如隐身Boss）中，相机不做特殊适配，而是依赖AOE技能命中来揭示Boss位置，这增加了战斗的紧张感

::ImageView{src="/images/placeholder/endfield-boss-camera.jpg" caption="Boss战中相机自动拉远，确保AOE范围指示器清晰可见" width="full"}
::

### 角色切换镜头

在战斗中切换主控角色时的相机处理：

- 切换角色时相机**不做剪切**，而是在约0.2-0.3s内平滑过渡到新角色的跟随位置
- 由于四名角色在场景中距离通常不会太远，过渡距离有限，视觉上较为自然
- 切换瞬间可能出现短暂的相机加速移动（"甩镜头"感），在快节奏战斗中偶尔造成些许方向感丢失
- 新版本中"切换角色"已非战斗核心操作（主要通过快捷键释放队友技能），因此切换频率大幅降低，相机切换的打断感也相应减弱

## 相机设计分析

### 设计取舍的逻辑

终末地的相机设计反映了一系列有意识的取舍：

**策略性 > 动作性**：
- 没有硬锁定系统、闪避不触发子弹时间、camera shake偏克制——这些选择统一指向一个方向：相机服务于策略判断，而非操作快感
- 战斗中的正反馈集中在连携技的"倾泻感"上，而非单次攻击的打击反馈

**信息传达的策略**：
- 相机负责提供主控角色周边的空间信息
- 更远距离和更全局的信息（队友状态、敌人位置、威胁方向）交给UI系统处理
- 这种"相机+UI"的协作模式是多角色同屏战斗的务实选择

**跨平台一致性**：
- 相机系统在PC和移动端保持了较高的一致性
- 移动端通过独立的触屏灵敏度设置来适配，而非改变相机行为逻辑

## 优缺点总结

### 优点
- 多角色同屏的**队友自动避让机制**设计巧妙，从AI行为层面解决遮挡问题，而非纯依赖相机
- 探索/战斗/建造三种模式的相机各有适配，模式间过渡平滑
- 战斗相机反馈风格与游戏定位匹配——克制的shake服务于清晰的策略判断
- 终结技/连携技的演出镜头时长控制得当，既有视觉冲击力又不过度打断战斗流
- 连携技连续触发时的快速特写形成了独特的"倾泻"节奏感

### 缺点
- 缺少硬锁定系统导致面对多目标时**精确控制感较弱**，在高难战斗中偶尔造成"打不到想打的目标"
- 狭窄空间（矿洞、工业管道区域）的碰撞处理仍有穿模和"贴脸"问题
- 战斗中角色切换时偶尔出现的"甩镜头"感可能造成短暂眩晕
- 闪避时缺乏任何相机反馈（如慢动作、zoom-out）使得完美闪避的**成就感薄弱**
- Boss战相机拉远幅度有时不够，大型Boss的全身攻击前摇偶尔超出视野
- 作为一款跨端游戏，移动端的相机操控精度天然受限于触屏输入

### 值得借鉴

1. **AI行为解决相机遮挡**：多角色同屏时让AI角色主动调整站位避免遮挡主控角色，这比调整相机位置更优雅——因为它不影响玩家对相机的控制预期
2. **演出时长的精确控制**：连携技的0.5-1秒快速特写是一个很好的平衡——让玩家获得反馈但不打断战斗节奏，特别是在频繁触发的场景下
3. **相机+UI协作传达信息**：承认相机无法在多角色场景中展示所有信息，将全局信息交给UI（伤害方向指示、状态图标等），是务实且有效的设计
4. **建造模式的视角分离**：为工业建造提供独立的半俯视自由视角，而非强迫TPS视角完成建造操作，大幅提升了建造体验的效率和舒适度
5. **策略游戏的反馈克制**：camera shake和演出镜头的克制使用避免了画面"过于吵闹"，让玩家能持续聚焦于技能轮转和资源管理的策略判断

## 与同类游戏的对比思考

### 与原神的对比
同为多角色切换的开放世界/箱庭游戏，但差异明显：
- 原神的角色切换是**战斗核心机制**（元素反应需要频繁换人），相机切换压力更大；终末地弱化了切换需求
- 原神没有AI队友同屏，相机只需追踪单个角色；终末地需要处理四人同屏的遮挡与构图
- 原神的战斗更偏动作，闪避有i-frame且反馈明确；终末地偏策略，闪避反馈刻意弱化

### 与战神的对比
两者看似相差甚远，但在"相机与UI协作"方面有相似思路：
- 战神因越肩紧跟视角导致大量盲区，通过威胁箭头UI弥补；终末地因多角色同屏信息量大，通过伤害方向指示和状态图标UI弥补
- 都选择了让相机保持稳定可控，将"缺失的信息"交给UI层处理，而非强制移动相机
